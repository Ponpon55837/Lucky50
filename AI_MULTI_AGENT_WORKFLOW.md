# 多代理開發協作流程

> 可移植至其他專案與 HackMD 的最小作業規範。

## 角色

| 角色 | 職責 |
| --- | --- |
| 總控／驗證者 | 接收需求、拆解任務、派工、驗證、決定下一步 |
| Agent 1／實作者 | 只修改指定範圍，完成實作與測試 |
| Agent 2／審查者 | 檢查 diff、規格、回歸、測試缺口；原則上不改檔 |

## 固定迴圈

```text
需求 → 總控拆解與定義驗收條件 → Agent 1 實作 → MCP 完成通知
     → Agent 2 交叉檢查 → MCP 審查通知 → 總控驗證
     → 通過：派下一批／收尾；不通過：退回 Agent 1 修正
```

總控必須主動推進流程，不要求使用者逐一操作 terminal 或 pane。

## 派工規則

每次派工都要寫清楚：

- 目標與非目標
- 指定檔案或範圍
- 驗收條件與測試指令
- 完成或阻塞時的 MCP 回報要求

Agent 不得擴大範圍；發現阻塞時立即回報，不可靜默停留。

## 標準回報格式

### Agent 1

```text
[agent-1 完成通知]
變更摘要：
檔案：
測試：
剩餘風險：
下一步建議：
```

### Agent 2

```text
[agent-2 審查通知]
結論：通過／需修正
問題與嚴重度：
檢查檔案或行號：
測試結果：
合併建議：
```

### 阻塞

```text
[agent 阻塞通知]
原因：
已嘗試：
需要的決策或外部條件：
```

## 總控驗證清單

- `git diff --check`
- 相關測試與全量測試
- TypeScript／build／lint（依專案工具鏈）
- 確認變更只在需求範圍
- UI 變更檢查 light/dark mode、響應式與無障礙
- 確認沒有未回報的風險或未完成任務

## 啟動範例

```bash
# 專案根目錄
npm run ai:bridge
```

建立三個 pane：總控、Agent 1、Agent 2；使用固定 session 名稱，例如 `ai-bridge`。啟動提示應明確要求 Agent 1／2 以 MCP 回報，總控收到後自動派發下一步。

## 無 MCP 時的 fallback

若 MCP 不可用，使用共享檔案作為事件佇列，例如 `.ai/status/`：

```text
.ai/status/agent-1.done.md
.ai/status/agent-2.review.md
```

總控輪詢檔案與 git 工作樹；Agent 仍須使用上述固定格式，確保流程可移植。

## 收尾格式

```text
結論：完成／部分完成／阻塞
變更：
驗證：
未處理項目：
下一步：
```
